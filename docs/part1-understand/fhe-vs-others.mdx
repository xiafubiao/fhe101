# FHE vs ZK vs MPC vs TEE

## 快速区分

在隐私保护技术领域，有四种主流方案常被拿来对比：

| 技术 | 全称 | 一句话理解 |
|------|------|-----------|
| **FHE** | Fully Homomorphic Encryption | 数据全程加密，云端可直接计算 |
| **ZK** | Zero-Knowledge Proof | 证明"我知道答案"而不泄露答案 |
| **MPC** | Multi-Party Computation | 多方数据联合计算，各方只拿到最终结果 |
| **TEE** | Trusted Execution Environment | 硬件层面的安全飞地，依赖物理信任 |

## 技术对比

### 信任假设

| 技术 | 信任模型 |
|------|----------|
| FHE | 依赖密码学假设（如 RLWE 问题） |
| ZK | 依赖密码学假设 |
| MPC | 多数诚实方假设或dishonest majority |
| TEE | 信任硬件厂商和 CPU 本身 |

### 计算能力

| 技术 | 支持的运算 |
|------|-----------|
| FHE | 密文加法、乘法、多项式运算 |
| ZK | 任意计算，但证明生成开销大 |
| MPC | 任意计算，取决于协议 |
| TEE | 任意明文计算（无运算限制） |

### 性能特征

| 技术 | 计算开销 | 通信开销 |
|------|----------|----------|
| FHE | 高（比明文慢 1000-100000x） | 低 |
| ZK | 证明生成慢，验证快 | 低 |
| MPC | 中等（多方通信轮数） | 高 |
| TEE | 低（接近裸机性能） | 低 |

## 何时用哪个？

### 选择 FHE 当你需要...
- 数据在加密状态下被云端/第三方处理
- 数据所有者不信任计算环境
- 场景：外包计算、隐私机器学习推理

### 选择 ZK 当你需要...
- 证明你知道某个答案而不透露答案本身
- 验证计算的正确性
- 场景：区块链隐私交易、身份验证、知识证明

### 选择 MPC 当你需要...
- 多方数据联合计算
- 各方只看到最终聚合结果
- 场景：联合风控、数据共享、阈值签名

### 选择 TEE 当你需要...
- 高性能通用计算
- 愿意信任硬件厂商
- 场景：机密计算、企业级安全飞地

## 组合使用趋势

现实应用中，这几种技术经常组合使用：

### FHE + ZK
可验证的同态计算：不仅能对密文计算，还能证明计算的正确性。

### FHE + MPC
提升性能或实现密钥分片：FHE 计算太慢？用 MPC 分担计算量。

### ZK + MPC
隐私 + 可验证：既保证多方数据不泄露，又能验证计算结果的正确性。

## 总结

没有最好的技术，只有最适合的技术。选择时考虑：
1. **你的威胁模型是什么？**
2. **需要保护什么？**
3. **性能预算有多少？**
