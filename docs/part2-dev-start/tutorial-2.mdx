# 案例二：区块链隐私交易

## 场景

在传统区块链上，所有的交易金额、余额都是公开的。假设 Alice 想向 Bob 转账，但她不想让别人知道自己有多少钱、转了多少。

使用 FHE 技术，可以在区块链上实现**机密 ERC-20 Token**，让交易金额和余额保持加密状态。

## 官方 TFHE-rs 示例

来源：[TFHE-rs Quick Start](https://docs.zama.ai/tfhe-rs/get-started/quick-start)

```rust
use tfhe::{ConfigBuilder, generate_keys, set_server_key, FheUint8};
use tfhe::prelude::*;

fn main() {
    let config = ConfigBuilder::default().build();

    // 客户端：生成密钥
    let (client_key, server_key) = generate_keys(config);

    let clear_a = 27u8;
    let clear_b = 128u8;

    // 客户端：加密数据
    let a = FheUint8::encrypt(clear_a, &client_key);
    let b = FheUint8::encrypt(clear_b, &client_key);

    // 服务端：在密文上计算
    set_server_key(server_key);
    let result = a + b;

    // 客户端：解密结果
    let decrypted_result: u8 = result.decrypt(&client_key);

    let clear_result = clear_a + clear_b;
    assert_eq!(decrypted_result, clear_result);
}
```

## 机密交易原理

```
传统 ERC-20 交易：
Alice 转账 100 USDC 给 Bob
→ 所有人可见：Alice 地址、Bob 地址、转账金额

机密 ERC-20 交易：
Alice 转账 E(100) USDC 给 Bob（E 表示加密）
→ 只有 Alice 和 Bob 知道实际金额
→ 验证者可以验证交易有效性，但不知道具体金额
```

## 与 ZK 的对比

| 特性 | FHE 方案 | ZK 方案（如 Tornado Cash） |
|------|----------|---------------------------|
| 隐藏内容 | 交易金额、余额 | 交易关联性（谁转给谁） |
| 透明性 | 交易本身公开，金额加密 | 交易公开，无金额信息 |
| 计算开销 | 高（密文运算） | 中等（证明生成） |
| 适用场景 | 保密金额的转账 | 匿名混币 |

## 关键点总结

1. **金额加密**：交易金额和余额在整个过程中保持加密
2. **可验证性**：验证者可以在不解密的情况下验证交易有效性
3. **选择性披露**：只有拥有私钥的用户才能解密查看实际金额
4. **合规性**：结合 ZK 可以实现监管可追溯性
