# FHE 核心概念

## 密钥体系

FHE 有三种密钥，理解它们是掌握 FHE 的基础：

```
┌─────────────────────────────────────────────────┐
│                  密钥体系                        │
├─────────────────────────────────────────────────┤
│  公钥 (Public Key)                              │
│    └── 用于加密数据，由私钥派生                 │
│                                                  │
│  私钥 (Secret Key / Client Key)                 │
│    └── 用于解密数据，用户自己保管               │
│                                                  │
│  计算钥 (Evaluation Key / Server Key)           │
│    └── 用于密文运算，云端持有                  │
└─────────────────────────────────────────────────┘
```

## 完整流程

### 步骤 1：密钥生成 (Key Generation)

```python
# 客户端生成密钥对
client_key = generate_client_key()
server_key = generate_server_key(client_key)

# 公钥发给云端（计算钥）
# 私钥自己保管，绝不泄露
```

### 步骤 2：加密 (Encryption)

```python
# 客户端加密敏感数据
plaintext = 42
ciphertext = encrypt(client_key, plaintext)

# 加密后的密文发给云端
send_to_cloud(ciphertext)
```

### 步骤 3：运算 (Evaluation)

```python
# 云端在密文上执行计算
# 注意：云端不知道 actual values
result = ciphertext_a + ciphertext_b  # 加法
result = ciphertext_a * ciphertext_b  # 乘法
```

### 步骤 4：解密 (Decryption)

```python
# 云端返回运算结果
response = receive_from_cloud(result)

# 客户端用私钥解密
final_result = decrypt(client_key, response)
# final_result = 实际运算结果
```

## FHE 运算原理

### 加密的本质：带噪声的密文

加密后的数据并不是"干净"的密文，而是包含了一层"噪声"：

```
密文 = 真实数据 + 噪声  (mod q)
```

以 LWE 加密为例：
```
b = a·s + Δm + e  (mod q)

其中：
- a: 随机公钥系数（向量）
- s: 私钥（向量）
- Δ = q/t: 明文缩放因子（t 是明文模数）
- m: 原始数据
- e: 噪声（通常是小随机数，通常 |e| < q/2）
- q: 密文模数（一个大整数）
```

> **为什么要有噪声？**
> 噪声是密码学安全性的基础。LWE 问题的安全性依赖于"从随机噪声中区分密文"这个数学难题。目前没有已知的多项式时间算法可以在量子计算机上高效解决此问题。

### 加法运算

两个密文相加，明文结果直接相加：

```
E(m₁) = (a₁, b₁)  其中 b₁ = a₁·s + Δm₁ + e₁
E(m₂) = (a₂, b₂)  其中 b₂ = a₂·s + Δm₂ + e₂

E(m₁) + E(m₂) = (a₁+a₂, b₁+b₂)
               = ((a₁+a₂)·s + Δ(m₁+m₂) + (e₁+e₂))
               = E(m₁ + m₂)
```

**特点**：
- 密文分量直接相加
- 明文结果 = m₁ + m₂
- 噪声累积：e' = e₁ + e₂（线性增长）
- 运算速度快，和普通整数加法相当

### 乘法运算

密文乘法比加法复杂得多，需要用到 **Gadget 分解**。

**问题**：如果直接相乘两个密文
```
b₁·b₂ = (a₁·s + Δm₁ + e₁)(a₂·s + Δm₂ + e₂)
       = a₁a₂·s² + Δm₁a₂·s + Δm₂a₁·s + Δ²m₁m₂ + ...(噪声项)
```

会出现 **s² 项**，这无法用原始密钥 s 解密。

**解决方案：Gadget 分解**

将大数值分解为小分量的线性组合：
```
Λ = Λ₁·(q/β₁) + Λ₂·(q/β₂) + ... + Λₗ·(q/βₗ)
   = Decomp_{β,ₗ}(Λ) · (q/β₁, q/β₂, ..., q/βₗ)ᵀ
```

其中 Decomp 是分解函数，每个 Λᵢ 是小整数（通常 < βᵢ）。

通过分解，可以把密文乘法的 s² 项转换为可管理的形式，然后用**密钥切换**消除。

**乘法结果**：
```
E(m₁) × E(m₂) = E(m₁ × m₂)
```

**特点**：
- 噪声乘性增长：e' ≈ m₁·e₂ + m₂·e₁ + e₁·e₂（比加法快得多地增长）
- 需要密钥切换（Key Switching）处理 s² 项
- 需要定期 Bootstrapping 清理噪声

### 密钥切换（Key Switching）

在不解密的情况下，将密文从密钥 s₁ 切换到 s₂：

```
GLWE_{s₁}(Δm + e)  --密钥切换-->  GLWE_{s₂}(Δm + e')
```

**用途**：
- 乘法后消除 s² 项
- 支持不同深度的电路级联
- 在不同密钥体系间转换密文

### 模切换（Mod Switching）

将密文模从 q 缩放到 q'（通常 q' < q）：

```
â = ⌈a·q'/q⌋  (mod q')
b̂ = ⌈b·q'/q⌋  (mod q')
```

**效果**：
- 噪声相对变小（噪声/q 的比例变化）
- 可以控制噪声和精度的权衡
- 用于层级式 FHE（BGV）减少密钥大小

## Bootstrapping 详解

### 为什么需要 Bootstrapping？

每次密文运算都会累积噪声。乘法尤其严重。

**问题**：噪声有上限
```
|e| < q/(2·t)   --解密正确
|e| ≥ q/(2·t)   --解密失败，得到错误结果
```

当噪声快耗尽时，必须"清理"噪声，否则无法继续运算。

这就像汽车油表：油量低于某个阈值时，必须去加油（Bootstrapping），否则无法继续行驶（计算）。

### Bootstrapping 原理

Bootstrapping = "在密文上执行解密操作"，本质是**同态解密**。

**核心思想**：
```
传统解密：在明文上计算 m = ⌊(b - a·s)/Δ⌋
同态解密：在密文上模拟这个计算过程
```

解密公式（以 LWE 为例）：
```
m = ⌊(b - a·s)/Δ⌋ (mod t)

这个公式涉及：
- 减法        -- 密文加法可以实现 ✓
- a·s        -- 密文×明文可以实现 ✓
- 四舍五入    -- 需要非线性操作 ✗ （难题！）
```

### 非线性操作的困难

四舍五入 ⌊x⌋ 是非线性的，FHE 只能做加法和乘法，无法直接实现。

**解决方案**：利用三角函数

```
⌊x⌋ ≈ (x - sin(2πx)) / (2π)
```

通过近似正弦函数，可以把取整操作转化为可计算的组合。

### TFHE 的 Gate Bootstrapping

TFHE 提出了一种巧妙方法，通过**盲旋转**实现任意门电路：

```
Gate Bootstrapping 两步法：

步骤 1：Blind Rotation（盲旋转）
─────────────────────────────
1. 输入 LWE 密文 (a, b) 和要评估的函数 f
2. 构造特殊多项式 g(X) = (b + Σ aᵢ·Xᵢ) mod (Xᴺ + 1)
3. 计算旋转结果：g(X)·X^(2^t · f(0))
4. 旋转变量 X 被旋转到正确位置，隐含了 f(m)

步骤 2：Coefficient Extraction（系数提取）
─────────────────────────────
1. 从旋转结果中提取目标系数
2. 恢复密文形式的解密结果 E(f(m))
```

> **直观理解**
> 把盲旋转想象成一个魔法盒子：
> - 输入：密文和要计算的函数
> - 内部：自动旋转到正确位置
> - 输出：函数计算结果的密文
>
> 整个过程在密文上完成，云端不知道函数是什么，也不知道输入是什么。

### Bootstrapping 的作用

| 作用 | 说明 |
|------|------|
| 噪声清理 | 噪声被重置到初始水平 |
| 任意运算 | 理论上可以评估任意深度电路 |
| 代价 | 计算开销大（可能比普通运算慢 1000x） |

### 各方案的 Bootstrapping 对比

| 方案 | Bootstrapping 速度 | 技术 | 特点 |
|------|-------------------|------|------|
| TFHE | **最快** | Gate bootstrapping | 专为快速自举优化 |
| BGV | 中等 | Digit extraction | 模切换技术成熟 |
| BFV | 较慢 | Digit extraction | 需要提取小数位 |
| CKKS | **最慢** | 近似正弦函数 | 需要高次多项式逼近 |

> **实际建议**
> - 浅层电路：可以不用 Bootstrapping（层级式 FHE）
> - 深层电路：必须用 Bootstrapping
> - TFHE 最适合需要频繁 Bootstrapping 的场景
> - CKKS 的 Bootstrapping 开销最大，谨慎使用

## 四大 FHE 方案对比

### 方案总览

| 方案 | 年份 | 基于 | 数据类型 | 精度 | 适用场景 |
|------|------|------|----------|------|----------|
| **TFHE** | 2018 | GLWE/LWE | 整数（位运算） | 精确 | 布尔电路、深度电路 |
| **BFV** | 2012 | RLWE | 整数 | 精确 | 整数运算、批处理 |
| **CKKS** | 2017 | RLWE | 复数/实数 | 近似 | 机器学习、信号处理 |
| **BGV** | 2012 | RLWE | 整数 | 精确 | 层级式、浅层电路 |

---

### TFHE（环面上的 FHE）

**核心特点**：
- 基于 GLWE 和 LWE
- 专为**布尔电路**设计
- Bootstrapping **最快**
- 适合**深度电路**场景

**加密结构**：
```
LWE 密文：(a₁, a₂, ..., aₙ, b)
          b = Σ(aᵢ·sᵢ) + Δm + e  (mod q)

每个系数都是模 q 下的整数
```

**适用场景**：
- 需要频繁 Bootstrapping
- 比较操作、任意逻辑门
- 神经网络激活函数（ReLU、MaxPool、Sigmoid）
- 隐私数据查询

**代表库**：TFHE-rs (Zama)、OpenFHE

---

### BFV（整数的批处理）

**核心特点**：
- 基于 RLWE
- **整数运算**，精确结果
- **批处理编码**：将多个整数打包到一个密文
- 适合**浅层电路**

**批处理编码**：
```
明文向量：[m₁, m₂, m₃, ..., mₙ]
                ↓ 批处理编码（ slot encoding）
单个密文：可以一次性对所有 mᵢ 做同态运算

优势：吞吐量提升 N 倍（N = poly_modulus_degree）
```

**适用场景**：
- 数据聚合、统计计算（求和、平均）
- 多项式求值
- 隐私数据库查询
- 固定深度电路

**代表库**：Microsoft SEAL、OpenFHE

---

### CKKS（近似计算的 FHE）

**核心特点**：
- **浮点数运算**，支持复数/实数
- 结果是**近似值**（有精度损失）
- **批处理**：对向量做并行运算
- 专为**机器学习**设计

**编码方式**：
```
复数向量 [z₁, z₂, ..., zₙ]
        ↓ CKKS 编码（放大、展平）
多项式 m(X)  --- 系数编码
        ↓ 加密
密文

解码时需要缩小回原来 scale
```

**精度控制**：
```
明文缩放因子 Δ = q/t
精度 ≈ log₂(Δ) 位
例如：Δ = 2⁴⁰ ≈ 1万亿，精度约 40 位十进制

噪声水平通常为 Δ/2 的量级
```

**适用场景**：
- 机器学习推理（矩阵乘法、卷积）
- 神经网络前向传播
- 信号处理
- 科学计算（微分方程求解）

**代表库**：HEAAN、SEAL、OpenFHE

**注意**：Bootstrapping 最慢，因为需要近似正弦函数评估。

---

### BGV（层级式 FHE）

**核心特点**：
- 基于 RLWE
- **层级式设计**：预设乘法深度 L
- **模切换**代替部分 Bootstrapping
- 适合**固定深度电路**

**与 BFV 区别**：
- BFV：需要 Bootstrapping 支持任意深度
- BGV：预设深度，用模切换控制噪声，无需 Bootstrapping 即可进行 L 次乘法

**适用场景**：
- 已知计算深度的应用
- 固定模型大小的 ML 推理
- 适合一次性部署固定深度的电路

**代表库**：OpenFHE、HElib

### 方案选择决策树

```
你需要什么类型的运算？
│
├─ 布尔/位运算（比较、逻辑门、ReLU）→ TFHE
│   └─ 适合深度电路，需要快速 Bootstrapping
│
├─ 整数运算（统计、聚合、加乘）→ BFV 或 BGV

│   ├─ 浅层电路，数据聚合 → BFV

│   │

│       └─ 无需 Bootstrapping，性能更优
│
└─ 浮点运算（ML推理、信号处理）→ CKKS
    └─ 近似计算，批处理效率高，适合神经网络
```

## 噪声与去中心化密钥管理

### 一、噪声（Noise）：FHE 的"计算预算"

#### 为什么会有噪声？

FHE 的安全性建立在 **LWE（Learning With Errors）** 问题的困难性上。简单说，加密时会故意往密文里注入一点随机噪声，让攻击者无法从密文中提取信息。

但这个噪声有个问题：**每次做同态计算，噪声都会增长**。

- 加法：噪声增长较小
- 乘法：噪声增长较大（大约翻倍）

当噪声超过一个阈值，解密就会出错——就像一张被涂改太多次的试卷，最终已经看不清原始答案。

#### 噪声直觉：把它理解为"计算预算"

```
新密文：          [噪声很小]  ████░░░░░░░░░░░░░░░░░░  安全边界
加一次加法后：    [噪声稍大]  ██████░░░░░░░░░░░░░░░░  安全边界
加一次乘法后：    [噪声较大]  ██████████░░░░░░░░░░░░  安全边界
再加一次乘法：    [噪声接近上限] ████████████████░░░░  安全边界
超出预算（错误）： [噪声溢出]  ████████████████████████ 安全边界（溢出！）
```

你可以把噪声想成**一桶水，每次运算都会往里加水，满了就溢出**。

#### 乘法深度（Multiplicative Depth）

你能做多少次乘法，取决于参数设置的"乘法深度"上限。这是 FHE 开发中最核心的约束之一：

```solidity
// fhEVM 里不需要手动管理噪声，
// Coprocessor 在链下自动处理，
// 但如果你用 TFHE-rs 直接开发，需要了解深度限制
```

#### Bootstrapping：给噪声"清零"

为了突破乘法深度的限制，Gentry 在 2009 年提出了 **Bootstrapping**：通过对密文本身进行同态解密，把噪声"刷新"到一个很小的值，从而可以继续计算。

代价是：Bootstrapping 是目前 FHE 里最慢的操作，一次可能需要几百毫秒到几秒。

**对 fhEVM 开发者**：好消息是 TFHE 的每次门操作后会自动 bootstrapping（这是 TFHE 的设计特点之一），所以你写 Solidity 合约时不需要手动管理。

### 二、密钥管理：去中心化环境的特殊挑战

#### 传统 FHE 的密钥模型

传统 FHE 是这样工作的：

```
用户  →  用公钥加密数据  →  服务器做同态计算  →  用户用私钥解密结果
```

问题在于：**私钥必须由用户自己持有**。这在中心化场景没问题，但在区块链上却带来了严重挑战——如果没有人持有私钥，谁来解密合约里的状态？

#### 去中心化环境下的困境

想象一个私密投票合约：

- 投票期间，余额应该保密（防止影响后续投票人）
- 投票结束后，结果需要被解密公布
- **但谁来解密？** 如果是合约部署者，他在投票期间就能偷看结果

这是一个典型的**信任假设问题**。

#### 解决方案：门限解密（Threshold Decryption）

fhEVM 的解决方案是用 **MPC + 门限密码学**：

```
                     ┌──────────────────────────────┐
                     │       密钥管理系统（KMS）        │
                     │                              │
用户                  │  节点A  节点B  节点C  节点D  │
 │                   │   sk₁    sk₂    sk₃    sk₄  │
 │  用全局公钥加密     │          （私钥分片）         │
 └──────────────────▶│                              │
                     │  解密需要 ≥ 3/4 节点共同参与   │
                     └──────────────────────────────┘
```

核心思想：
1. **全局公钥**：所有人用同一个公钥加密数据，这个公钥是公开的
2. **私钥分片**：私钥被分成多份，由多个独立节点持有，任何单个节点都无法独立解密
3. **门限解密**：只有足够多的节点（如 3/4）共同参与，才能解密结果

这样就实现了：**没有任何单点能够偷看加密数据，但合法的解密请求可以被满足**。

#### 在 fhEVM 里的具体表现

fhEVM 的 ACL（访问控制列表）系统管理"谁可以请求解密"：

```solidity
// 允许合约自身访问该密文
FHE.allowThis(encryptedValue);

// 允许特定地址（如用户）在链下解密查看自己的数据
FHE.allow(encryptedValue, msg.sender);

// 解密不是在链上发生的——而是通过 Gateway 协调 KMS 节点完成
```

#### 开发者需要知道的

作为 fhEVM 开发者，你不需要直接实现门限密钥管理。但你需要理解：

1. **所有加密数据共用一个全局公钥**：不同用户的密文可以在合约里互相运算（因为用同一把公钥加密）
2. **解密是链下发生的**：合约请求解密后，Gateway 协调 KMS 节点，结果异步返回
3. **ACL 控制解密权限**：你必须显式授权谁可以看到哪个密文的明文

---

### 三、密文的"体积"问题

FHE 密文比明文大得多，这会影响你的合约设计：

| 数据 | 明文大小 | FHE 密文大小（参考） |
|------|----------|---------------------|
| uint32（余额） | 4 bytes | ~1-4 KB |
| 一个布尔值 | 1 bit | ~1 KB |

**实际影响**：
- 链上存储成本更高（但密文通常存储在链下，链上只存句柄 handle）
- 传输加密输入时，用户需要传递更大的数据包

在 fhEVM 里，你操作的是**密文句柄（ciphertext handle）**，而不是实际密文，链上存储的是句柄（一个 32 字节的引用），实际密文由 Coprocessor 管理。
