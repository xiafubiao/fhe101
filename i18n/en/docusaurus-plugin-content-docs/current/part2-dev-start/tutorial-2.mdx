# Example 2: Blockchain Privacy Transactions

## Scenario

In traditional blockchains, all transaction amounts and balances are public. Suppose Alice wants to transfer money to Bob, but she doesn't want others to know how much she has or how much she transfers.

Using FHE technology, it's possible to implement **Confidential ERC-20 Tokens** on the blockchain, keeping transaction amounts and balances encrypted.

## Official TFHE-rs Example

Source: [TFHE-rs Quick Start](https://docs.zama.ai/tfhe-rs/get-started/quick-start)

```rust
use tfhe::{ConfigBuilder, generate_keys, set_server_key, FheUint8};
use tfhe::prelude::*;

fn main() {
    let config = ConfigBuilder::default().build();

    // Client-side: generate keys
    let (client_key, server_key) = generate_keys(config);

    let clear_a = 27u8;
    let clear_b = 128u8;

    // Client-side: encrypt data
    let a = FheUint8::encrypt(clear_a, &client_key);
    let b = FheUint8::encrypt(clear_b, &client_key);

    // Server-side: compute on ciphertext
    set_server_key(server_key);
    let result = a + b;

    // Client-side: decrypt result
    let decrypted_result: u8 = result.decrypt(&client_key);

    let clear_result = clear_a + clear_b;
    assert_eq!(decrypted_result, clear_result);
}
```

## Confidential Transaction Principles

```
Traditional ERC-20 transaction:
Alice transfers 100 USDC to Bob
→ Visible to everyone: Alice address, Bob address, transfer amount

Confidential ERC-20 transaction:
Alice transfers E(100) USDC to Bob (E means encrypted)
→ Only Alice and Bob know the actual amount
→ Verifiers can verify transaction validity without knowing the amount
```

## Comparison with ZK

| Feature | FHE Solution | ZK Solution (e.g., Tornado Cash) |
|---------|-------------|----------------------------------|
| Hidden Content | Transaction amounts, balances | Transaction relationships (who sent to whom) |
| Transparency | Transaction itself is public, amount encrypted | Transaction is public, no amount info |
| Computation Cost | High (ciphertext operations) | Medium (proof generation) |
| Use Cases | Transfers with confidential amounts | Anonymous mixing |

## Key Takeaways

1. **Amount encryption**: Transaction amounts and balances remain encrypted throughout
2. **Verifiability**: Verifiers can verify transaction validity without decrypting
3. **Selective disclosure**: Only users with the private key can decrypt to view actual amounts
4. **Compliance**: Combined with ZK, it can achieve regulatory traceability
